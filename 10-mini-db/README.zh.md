# 10 — mini-db

> 一个最小化的内存 SQL 数据库引擎：用 Python 展示 tokenizer、语法解析、`WHERE` 布尔表达式求值，以及基于有序列表的索引。

## 背景

哪怕是一个非常小的 SQL 数据库，也至少要解决三件事：

- 把 SQL 文本切成 token
- 把语句解析成可执行的结构
- 在表数据上执行过滤、插入、删除等操作

工业级数据库还会继续加上事务、锁、缓冲池、磁盘页、查询优化器、日志恢复等复杂机制。这个项目故意不做那些重量级部分，而是聚焦在“SQL 是如何一步步被执行起来的”。

## 架构

### 1. Tokenizer + Parser

实现首先用一个正则把 SQL 切成：

- 标识符
- 数字
- 字符串
- 标点和比较符号

然后用一组很短的递归下降解析函数，支持：

- `CREATE TABLE`
- `CREATE INDEX`
- `INSERT INTO`
- `SELECT ... FROM ... WHERE ...`
- `DELETE FROM ... WHERE ...`

这样就形成了一个足够小、但足够完整的 SQL 子集。

### 2. In-memory Table Storage

每张表都维护三部分：

- 列定义
- 行列表（每行是一个 Python `dict`）
- 可选索引

这个结构非常直接，优势是执行器写起来很短：扫描一行，拿列名取值，求值表达式，决定是否保留结果。

### 3. 索引结构：用有序列表模拟 B-tree 思路

题目允许使用真正的 B-tree，或者用更简单的 sorted-list simulation。这里选择后一种：

- `CREATE INDEX` 会把某一列的值整理成有序 `(value, row_id)` 列表
- 查询 `WHERE age = 28` 这类等值条件时
- 用 `bisect` 直接定位匹配区间

它不是完整 B-tree，但保留了 B-tree 最重要的收益：**数据按 key 有序组织，查询不必总是全表扫描。**

## 关键实现分析

### `WHERE` 子句如何解析

`WHERE` 表达式支持：

- `=`
- `>`
- `<`
- `AND`
- `OR`

解析器按优先级拆成：

1. `OR`
2. `AND`
3. comparison

因此像 `city = 'London' OR age < 30 AND age > 20` 会按标准布尔优先级求值，而不是简单从左往右乱算。

### 执行器如何工作

对于普通查询，执行器就是：

1. 拿到目标表
2. 遍历每一行
3. 对每行执行 `WHERE` 表达式
4. 把命中的行投影成 `SELECT` 需要的列

`DELETE` 也类似，只不过最后留下未命中的行，并返回删除数量。

### 索引加速为什么只做等值查询

这个实现里，索引优化只在如下场景触发：

- 顶层条件是单个 comparison
- 操作符是 `=`
- 对应列存在 index

这是一个刻意的裁剪，因为教学重点是展示“索引如何改变访问路径”，而不是把 planner 做复杂。即使只支持等值查找，也已经能很直观地看到索引命中的效果。

## 如何运行

```bash
python3 demo.py
```

`demo.py` 会演示：

- 建表
- 插入多行数据
- 创建索引
- 带 `WHERE` 的查询
- 使用索引的等值查询
- 删除满足条件的行

## 关键收获

- SQL 引擎最核心的流水线就是：tokenize -> parse -> execute。
- 一个很小的 AST 加上一组行字典，就足以解释查询执行的本质。
- 即使只是 sorted-list index，也能清楚展示 B-tree 类结构为什么能加速查找。
