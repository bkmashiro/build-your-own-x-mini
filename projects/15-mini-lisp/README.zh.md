# mini-lisp

> 纯 Python 实现的极小 Lisp 解释器：词法切分、reader、求值器、闭包和尾调用优化。

[English](README.md)

---

## 背景

Lisp 是最适合用来理解解释器结构的“小语言”之一：

- 先把源码切成 token
- 再把 token 读成嵌套列表
- 然后在环境链里对列表求值
- 函数本质上就是捕获定义时作用域的闭包

因此它很适合做一个 200 行以内的实现。它的语法非常小，是因为“代码就是数据”：函数调用本身就是一个列表，第一个元素是操作符。

---

## 架构

```text
源码
  -> tokenize
  -> 解析成嵌套 Python 列表
  -> 在链式环境里求值
  -> 调用内建函数或用户自定义过程
```

这个解释器支持：

- 数字和布尔值
- `define`、`lambda`、`if`、`begin`、`quote`
- 基于词法作用域的闭包
- 针对用户函数的尾调用优化

---

## 关键实现

### Reader

Reader 会把括号表达式转成嵌套 Python 列表。例如：

```lisp
(+ 1 (* 2 3))
```

会变成：

```python
["+", 1, ["*", 2, 3]]
```

### 环境

每个作用域都是一个 `Env`，并指向它的外层作用域。变量查找时沿着外层链一路向上，因此可以很自然地得到词法作用域。

### 闭包

`lambda` 会返回一个 `Procedure`，里面保存参数列表、函数体，以及它定义时所在的环境。调用时只需要创建一个新作用域，并把外层指针指回这个捕获环境即可。

### 尾调用

求值器在调用用户函数时不会总是递归进入 `evaluate`，而是通过循环改写当前的 `expr` 和 `env`。因此像 `sumdown` 这样的尾递归程序可以跑很多步而不会不断增长 Python 调用栈。

## 运行方式

```bash
python projects/15-mini-lisp/demo.py
python -m pytest projects/15-mini-lisp/test -q
```

demo 会构造一个闭包，执行一个尾递归求和，并打印最终得到的列表。

## 这里省略了什么

- 宏系统
- 字符串和注释
- quasiquote / unquote
- 多表达式 lambda 函数体
- 垃圾回收细节

这些属于更完整的 Scheme / Lisp 实现，但理解 read-eval-apply 主循环并不需要它们。
